# Архитектура проекта FictionBook Library

## 1. Основные принципы

| Принцип | Описание |
|---|---|
| **Unified Domain Model** | Единая модель (DTO), не знающая о формате (FB2/FB3) |
| **Immutable DTO** | Java Records — потокобезопасность, предсказуемость |
| **Прощающее чтение, строгая запись** | Читаем "как есть", пишем "по стандарту" (через санитайзеры) |
| **Гибридный парсинг** | Jackson для `<description>`, StAX для `<body>` |
| **Stack of Builders** | Стек билдеров для парсинга mixed content в `<body>` |
| **Eager-загрузка бинарников** | Base64 декодируется сразу при чтении (проще, чем seek) |
| **Event-driven рендеринг** | Command Pattern — книга "проигрывает" себя на рендерер |
| **Separation of concerns** | DTO не содержит индексов/кэшей — это производные структуры |

## 4. Стек технологий

| Компонент | Выбор | Почему |
|---|---|---|
| **Java** | 21+ | Records, Pattern Matching, Virtual Threads |
| **XML парсинг (мета)** | Jackson XML | Удобный маппинг для структурированных данных |
| **XML парсинг (тело)** | StAX (Woodstox) | Быстро, мало памяти, mixed content |
| **ZIP (FB3)** | `java.util.zip` | Встроен в JDK, достаточно |
| **Кодировки** | juniversalchardet | Проверенный Mozilla-детектор |
| **Логирование** | SLF4J API | Без навязывания реализации |
| **Тесты** | JUnit 5 + AssertJ + jqwik | Round-trip + Property-based + Fuzz |
| **Бенчмарки** | JMH (`me.champeau.jmh`) | read / write / sanitize / round-trip |

## 5. Ключевые архитектурные решения

### 5.1. Два режима работы

```
┌─────────────────────────────────────────────────────────┐
│  Fb2Reader (DOM)                                        │
│  ─────────────────                                      │
│  • Читает всю книгу в FictionBookDto                    │
│  • Бинарники загружаются сразу (eager base64 decode)    │
│  • Для конвертации, редактирования, индексации          │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│  Fb2Streamer (Streaming)                                │
│  ─────────────────────────                              │
│  • Читает поглавно, минимальное потребление RAM         │
│  • Для читалок: рендеринг на лету                       │
│  • AnchorIndex с byte offset для seek                   │
└─────────────────────────────────────────────────────────┘
```

**Почему eager, а не lazy для бинарников?**
Ленивое чтение через `RandomAccessFile.seek()` требует точных byte offset'ов в исходном файле (до перекодировки), синхронизации с `BufferedInputStream`, учёта различий в длине символов между кодировками. Eager-загрузка проще и надёжнее, а бинарники в FB2 обычно небольшие (50-500 KB).

### 5.2. Абстракция FB2/FB3

```
FictionBookDto (единая модель)
       ▲
       │
  ┌────┴────┐
  │         │
Fb2Reader  Fb3Reader (todo)
(XML)      (ZIP + XMLs)
```

FB3 — это ZIP-контейнер, внутри которого те же XML-структуры. Ядро парсера будет переиспользоваться для обоих форматов.

### 5.3. Пайплайн обработки книги

```
┌──────────┐     ┌────────────┐     ┌──────────────┐     ┌──────────┐
│  Файл    │────>│  Fb2Reader │────>│FictionBookDto│────>│  Модиф.  │
│ (FB2/FB3)│     │ (Jackson + │     │ (immutable)  │     │          │
└──────────┘     │   StAX)    │     └──────────────┘     └────┬─────┘
                 └────────────┘                               │
                                                              ▼
┌──────────┐     ┌────────────┐     ┌──────────────┐     ┌──────────┐
│  Файл    │<────│ Fb2Writer  │<────│ Sanitizer    │<────│  Новая   │
│ (FB2/FB3)│     │ (StAX)     │     │  Pipeline    │     │   копия  │
└──────────┘     └────────────┘     └──────────────┘     └──────────┘
```

### 5.4. AnchorIndex — производная структура

**Ключевой принцип:** `FictionBookDto` — это чистые данные, без производных индексов. Индекс строится по требованию.

```java
// DTO НЕ содержит индекс
FictionBookDto book = FictionBookIO.read(file);

// Индекс строится, когда нужен (санитайзеры, рендереры, стример)
AnchorIndex index = AnchorIndexBuilder.fromDto(book);
```

### 5.5. Event-driven рендеринг

```
FictionBookDto ──▶ BookPlayer ──▶ FictionBookRenderer ──▶ HTML / Text / JavaFX
                   (обход дерева)   (Command Pattern)
                                         ▲
                                    ResourceResolver
                                    (картинки)
```

- **BookPlayer** инкапсулирует логику обхода дерева и контекст (стек стилей).
- **FictionBookRenderer** — это набор команд (`startBold`, `text`, `startLink`, etc.).
- **ResourceResolver** — стратегия обработки картинок (base64, save to disk, placeholder).

### 5.6. Санитизация перед записью

`SanitizerPipeline` — цепочка санитайзеров, применяемых перед записью:

1. **EmptyParagraphCleaner** — удаление пустых параграфов
2. **EmptySectionCleaner** — удаление пустых секций
3. **TextNodeMerger** — склейка разбитых Text-нод
4. **OrphanedImageCleaner** — проверка ссылок на картинки
5. **OrphanedLinkCleaner** — проверка внутренних ссылок
6. **AttributeNormalizer** — приведение id к NCName

Все санитайзеры:
- Immutable (возвращают новую копию DTO)
- Идемпотентные (повторный запуск не вносит изменений)
- Безопасные (ловят исключения, логируют, продолжают)

## 6. Что уже работает end-to-end

### Полный цикл чтения-записи
```java
FictionBookDto book = FictionBookIO.read(Path.of("input.fb2"));
// ... модификации через FictionBookDtoTransformer ...
FictionBookIO.write(book, Path.of("output.fb2"));
```

### Рендеринг в HTML
```java
FictionBookDto book = FictionBookIO.read(file);
HtmlRenderer renderer = HtmlRenderer.builder()
    .wrapInHtmlDocument(true)
    .title(book.description().titleInfo().bookTitle())
    .resourceResolver(ResourceResolver.saveToDirectory(imagesDir, "images/"))
    .build();

BookPlayer player = new BookPlayer(renderer, href -> lookupResource(book, href));
player.play(book);
Files.writeString(outputHtml, renderer.getOutput());
```

### Извлечение plain text
```java
PlainTextRenderer renderer = new PlainTextRenderer();
new BookPlayer(renderer).play(book);

String text = renderer.getOutput();
int words = renderer.getWordCount();
String preview = renderer.getPreview(500);
```

## 8. Принятые архитектурные решения (ADR)

### ADR-001: Eager-загрузка бинарников вместо lazy
**Контекст:** FB2-файлы содержат картинки как base64 внутри `<binary>`.
**Решение:** Декодировать base64 сразу при чтении.
**Обоснование:** Lazy-чтение через `RandomAccessFile.seek()` требует сложных offset'ов и не работает с перекодированными потоками. Бинарники обычно небольшие.

### ADR-002: DTO не содержит AnchorIndex
**Контекст:** Для валидации ссылок нужен индекс якорей.
**Решение:** `AnchorIndex` строится отдельно через `AnchorIndexBuilder.fromDto()`.
**Обоснование:** DTO должно быть "плоским" (чистые данные). Индекс нужен не всегда, и его построение имеет свою стоимость.

### ADR-003: Jackson для description, StAX для body
**Контекст:** FB2 имеет mixed content в `<body>` и структурированные данные в `<description>`.
**Решение:** Гибридный подход.
**Обоснование:** Jackson не справляется эффективно с mixed content и потребляет много памяти на больших файлах. StAX быстрый, но требует boilerplate.

### ADR-004: Event-driven рендеринг
**Контекст:** Нужны разные рендереры (HTML, PlainText, JavaFX, PDF).
**Решение:** Command Pattern — книга "проигрывает" себя на рендерер.
**Обоснование:** Позволяет инкапсулировать логику обхода дерева в `BookPlayer`, а рендереры делать простыми.

### ADR-005: Санитизация перед записью
**Контекст:** FB2 файлы из интернета часто "грязные".
**Решение:** Принцип "прощающее чтение, строгая запись".
**Обоснование:** При чтении допускаем любые отклонения, при записи приводим к стандарту через цепочку санитайзеров.

### ADR-006: Stack of Builders для mixed content
**Контекст:** `<p>`, `<v>` и другие контейнеры содержат смешанный контент (текст + вложенные теги `<strong>`, `<a>`, `<image>` в произвольном порядке). Классический `switch` по событиям StAX превращается в спагетти-код с вложенными флагами.
**Решение:** Интерфейс `NodeBuilder` + стек (`Deque<NodeBuilder>`). При встрече открывающего тега билдер кладётся в стек, при тексте — добавляется в верхний билдер, при закрывающем — `build()` и результат передаётся родителю.
**Обоснование:**
- Инкапсуляция логики каждого тега в своём классе (`ParagraphBuilder`, `InlineContainerBuilder<T>`, `LinkBuilder`, `ImageBuilder`, `IgnoreBuilder`)
- StAX-специфика: `XMLStreamReader` бьёт текст на куски — `StringBuilder` внутри билдеров корректно их склеивает
- Прощающий режим: `IgnoreBuilder` для неизвестных тегов просто поглощает содержимое
- Расширяемость: добавление нового тега = один класс + одна строка в `switch`

### ADR-007: Отключение IS_NAMESPACE_AWARE
**Контекст:** FB2 использует namespace'ы (`http://www.gribuser.ru/xml/fictionbook/2.0`, `http://www.w3.org/1999/xlink`), но в реальных файлах префиксы могут отличаться (`l:`, `xlink:` и т.д.).
**Решение:** `XMLInputFactory.IS_NAMESPACE_AWARE = false`. Атрибуты читаются через универсальный `getAttributeValue(xml, localName)`, который проверяет namespace URI, имя без namespace и имя с префиксом.
**Обоснование:** Упрощает сравнение тегов (не нужно проверять namespace URI для каждого тега) и делает парсер устойчивым к разным префиксам.
